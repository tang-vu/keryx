> **Proof model:** Settlements use Circle Gateway batching — each payment returns a Circle facilitator
> settlement ID (a UUID, **not** a per-tx EVM hash), and many nanopayments batch into a few on-chain
> `submitBatch` txs (Arc, <500ms BFT finality). Verifiable on-chain proof is therefore the **batched
> settlement wallet** + the **SourceRegistry contract** (`0x2e12Fa…`) on testnet.arcscan.app — not a
> per-payment `/tx/` link. All figures here are real settled payments only.
>
> **Honesty on usage:** most volume is generated by Keryx's own autonomous agents — the volume engine
> plus a 24/7 headless web/A2A client — so it is real on-chain settlement but agent-driven, not yet
> majority external human traffic. The dashboard now labels web/A2A vs engine separately (the "by entry
> path" split above). Real external on-ramps exist (the MCP server on npm, the A2A `/api/agent/ask`
> endpoint, and the owner-verified conzit.com creator); driving genuine third-party usage is the active
> next push.

## How we generate genuine volume (not vanity)
The seed/volume engine (`scripts/seed-engine.mts`) runs the *real* agent making *real* autonomous
buy/skip/cache decisions over a stream of real questions — every payment is a settled x402 nanopayment
to a real creator wallet. Budget-guarded so spend is bounded. This shows "real volume" without needing
thousands of human users, while every transaction is a genuine autonomous agent decision.
